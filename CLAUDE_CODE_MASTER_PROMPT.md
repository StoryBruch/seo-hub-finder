# CLAUDE CODE MASTER PROMPT — Build SEO Hub Finder

Copy this entire prompt into Claude Code.

---

You are building a small but polished Python prototype called **SEO Hub Finder**.

The goal is to analyze Google Search Console exports and discover Programmatic SEO opportunities automatically.

The most important requirement:

> Do not make the user manually define every Programmatic SEO pattern.

The tool must **not** rely on a static hardcoded list like:

- `beste_x_fuer_y`
- `brand_reinigen`
- `brand_entkalken`
- `x_rezept`
- `x_alternative`

Those examples may appear in documentation and sample data, but the main discovery logic must infer repeatable structures from the GSC queries themselves.

Build a fast, almost finished prototype with:

1. A CLI script.
2. A Streamlit demo app.
3. Sample data.
4. A README.
5. A detailed search-volume validation workflow.
6. Clean CSV, Markdown, ZIP and HTML outputs.

The prototype must work locally without paid APIs.

---

## Product flow

The product flow is:

1. User uploads/imports a Google Search Console CSV.
2. The tool normalizes columns.
3. The tool filters for ranking proof.
4. The tool automatically discovers repeatable query structures.
5. The tool scores whether each structure is programmatic/scalable.
6. The tool creates a keyword-volume check queue.
7. User checks those keywords in an external keyword-volume tool, preferably Google Keyword Planner or any CSV-based keyword tool.
8. User re-imports the exported volume CSV.
9. The tool outputs only volume-confirmed opportunities.
10. The tool generates a content hub plan, article templates and internal linking recommendations.

---

## Core concept

GSC is the source for ranking proof.

Search volume must come from a separate keyword tool.

The tool must separate:

- discovered GSC patterns
- unvalidated keyword candidates
- volume-confirmed opportunities
- rejected keywords/patterns with reason
- final content hubs

This distinction is crucial. Do not present a full hub recommendation unless the pattern has at least some volume-confirmed keywords.

## Ranking proof filter

A query qualifies as having ranking proof when at least one condition is true:

```txt
position <= top_position
clicks > 0
impressions >= min_gsc_impressions AND position <= expanded_position
```

Default values:

```txt
top_position = 10
expanded_position = 20
min_gsc_impressions = 20
```

## Automatic pattern discovery

Do not use a hardcoded SEO pattern list.

Instead:

1. Normalize query text.
2. Tokenize each query.
3. Count token frequency across ranking-proof queries.
4. Detect repeated anchors, prefixes, suffixes and connectors.
5. Replace variable spans with slots:

```txt
{slot_1}
{slot_2}
{slot_3}
```

Examples from data:

```txt
delonghi magnifica entkalken
jura e8 entkalken
saeco xelsis entkalken
```

should produce:

```txt
{slot_1} entkalken
```

Another example:

```txt
beste kaffeebohnen für vollautomat
beste bohnen für cappuccino
beste kaffee für latte macchiato
```

should produce something like:

```txt
beste {slot_1} für {slot_2}
```

Another example:

```txt
flat white rezept
cold brew rezept
espresso tonic rezept
```

should produce:

```txt
{slot_1} rezept
```

These are not hardcoded rules. They are expected outcomes of the pattern discovery logic.

## Pattern scoring

Each pattern candidate should be scored with:

```txt
query_count
slot_diversity
top10_count
total_clicks
total_impressions
avg_position
confidence
```

Default thresholds:

```txt
min_pattern_queries = 3
min_distinct_slot_values = 3
min_template_confidence = 0.45
```

A pattern should be accepted before volume validation only when:

```txt
query_count >= min_pattern_queries
slot_diversity >= min_distinct_slot_values
confidence >= min_template_confidence
```

Rejected patterns must still be exported with `reject_reason`.

## Search volume validation gate

This is mandatory.

If no volume CSV is uploaded:

- create `keyword_volume_check_queue.csv`
- show pattern candidates
- do not present the hub as final
- mark opportunities as `needs_volume_validation` or `not_recommended_yet`

If volume CSV is uploaded:

- merge by normalized keyword/query
- add `search_volume`, `competition`, `cpc`
- accept only keywords with `search_volume >= min_volume`
- default `min_volume = 10`
- make `min_volume` editable in Streamlit and configurable in CLI

Final keyword statuses:

```txt
confirmed_opportunity
not_recommended_yet
rejected_low_or_missing_volume
```
